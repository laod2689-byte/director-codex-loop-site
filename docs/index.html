Agent Orchestration / Review Loop

Director-Codex Loop

一个把“需求理解、任务拆解、代码执行、测试验证、结果审查”收敛进同一闭环的双 Agent 工程系统。 它不是单轮问答，而是可追溯、可迭代、可验收的 AI 编排工作流。

Claude 负责规划与验收 Codex 负责落地与修改 测试结果与 git diff 自动回流 支持多轮闭环修正
核心痛点 人工串联开发流程慢，验收标准容易丢失
核心能力 长链推理 + 多 Agent 协作 + 自动校验回路
本地证据 已保留多轮运行产物与审批结果

项目解决的核心痛点

传统开发里，需求分析、约束提取、执行修改、测试、审查通常分散在多个工具和多轮沟通里， 既慢又容易出现“实现了功能，但没满足验收条件”的情况。这个项目把这些环节编排成一个自动闭环， 让 AI 不只是生成代码，而是持续承担规划、执行、复核和迭代修正的角色。

适用场景

  • 已有仓库中的定向功能开发与补测
  • 需要显式验收标准的代码交付任务
  • 需要保留执行证据和审查记录的 AI 工程流程
  • 适合验证多 Agent 编排是否优于单模型直出

核心逻辑流

Step 1 任务接收

输入自然语言目标、仓库路径和测试命令。

Step 2 Director 规划

Claude 提炼约束、列出假设、定义验收标准与风险。

Step 3 Worker 执行

Codex 直接在目标仓库中修改代码或生成文件。

Step 4 自动采证

控制器收集测试结果、执行日志和 git diff。

Step 5 审查裁决

Claude 根据证据判断是否通过，或返回具体整改意见。

Step 6 闭环迭代

未通过则进入下一轮，直到满足验收标准为止。

为什么它属于长链推理

这个系统不是一次性产出结果，而是要求模型先完成目标理解、范围界定、验收设计，再进入执行和验证阶段。 只有当“代码变更 + 测试结果 + 差异证据”共同满足条件时才会批准，因此整个过程天然带有跨步骤、跨状态的链式推理。

为什么它属于多 Agent 协作

Director 和 Worker 的职责是硬拆分的：前者不写代码，只负责计划和审查；后者不做抽象管理，只负责按计划落地。 这种分工降低了“同一个模型既当裁判又当运动员”的偏差，也让整个系统更适合扩展为更复杂的 Agent 网络。

本地已验证的成果

  • 在 smoke repo 中完成过代码修复闭环，最终测试命令 python -m pytest -q 通过，状态为 approved。
  • 在另一轮真实任务中完成过 2 轮编排闭环，自动生成了 51-watering-system/main.c，并通过结构化审查批准。
  • 系统会保留完整运行工件，包括 director_plan.json、worker_report、test_result、git_diff、director_review 和 result.json。

运行说明

python director_loop.py "实现登录接口并补齐测试" --test "pytest -q"
